# 钉钉进展汇报

## 核心原则

1. **阶段性汇报** - 仅在关键里程碑完成时推送，避免频繁打扰
2. **标题明确** - 使用标准化前缀快速识别消息类型
3. **内容精炼** - 突出重点，省略细节

## 汇报时机（严格遵守）

| 阶段 | 触发条件 | 标题前缀 |
|------|----------|----------|
| 需求分析 | PRD文档完成 | 📋 [需求分析] |
| 设计 | 架构/技术方案确定 | 🏗️ [设计完成] |
| 分工 | 任务拆解并分配完毕 | 👥 [任务分工] |
| 开发 | 核心功能模块完成 | 💻 [开发进展] |
| 测试 | 测试用例执行完毕 | 🧪 [测试报告] |
| Bug修复 | 修复重要/阻塞性Bug | 🐛 [Bug修复] |
| 交付 | 项目上线/交付 | 🚀 [项目交付] |

## 消息格式模板

```
[前缀] 项目名称

✅ 完成内容（1-3条要点）
📌 下一步计划（可选）
⚠️ 风险/阻塞项（可选）
```

## 使用方式

执行推送脚本：

```bash
./scripts/notify.sh "消息内容"
```

### 示例

```bash
# 需求分析完成
./scripts/notify.sh "📋 [需求分析] 用户管理模块

✅ 完成功能点梳理：注册/登录/权限管理
✅ 确定技术选型：JWT认证
📌 下一步：进入架构设计"

# 开发里程碑
./scripts/notify.sh "💻 [开发进展] 用户管理模块

✅ 完成用户注册API
✅ 完成登录鉴权逻辑
📌 剩余：权限管理模块（预计2天）"

# Bug修复
./scripts/notify.sh "🐛 [Bug修复] 登录模块

✅ 修复：Token过期未正确刷新
✅ 影响范围：所有需登录接口
⚠️ 建议：前端同步更新Token处理"
```

## 汇报频率控制

- **同一阶段**：最多汇报1次（如开发阶段仅在重要里程碑汇报）
- **Bug修复**：仅P0/P1级别Bug修复后汇报
- **日常沟通**：不通过此渠道，使用其他即时通讯

## 自动集成

在使用 `elite-dev-team` skill进行项目开发时，以下节点自动触发汇报：

1. 产品经理完成PRD → 📋 [需求分析]
2. 架构师完成设计文档 → 🏗️ [设计完成]
3. Tech Lead完成任务分解 → 👥 [任务分工]
4. 开发完成核心模块 → 💻 [开发进展]
5. QA完成测试报告 → 🧪 [测试报告]
6. 修复阻塞性Bug → 🐛 [Bug修复]
7. DevOps完成部署 → 🚀 [项目交付]
